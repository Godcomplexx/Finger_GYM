## Пункт 3. Проектирование архитектуры модуля

**Цель:** разработать архитектурную схему, достаточную для распределения задач между разработчиками и оценки трудозатрат.

**Логические слои:**

- **Initialization** — запуск камеры, проверка освещения и качества изображения, определение наличия руки в кадре.
- **Calibration** — калибровка по открытой ладони: вычисление `palmCenter`, `palmWidth`, базовых нормированных расстояний.
- **Exercise Engine** — управление последовательностью 8 заданий: вывод инструкции, ожидание выполнения, фиксация результата, переход к следующему заданию.
- **Tracking Adapter** — обёртка над SDK трекинга, возвращает 21 ключевую точку кисти и флаг достоверности кадра.
- **Data Processing** — нормировка координат на `palmWidth`, расчёт `fingerCurl`, `tipToPalmDistance`, `thumbIndexDistance`, `handJitter`, `holdTime`.
- **Scoring Engine** — расчёт частных баллов по каждому заданию, расчёт `validTrackingRatio`, итоговый индекс готовности по таблице весов из методики.
- **Recommendation Engine** — перевод индекса в одну из четырёх рекомендаций: стандартный режим, адаптированный, тренировочный, повтор тестирования.
- **Presentation** — экран пациента (инструкции, таймер, индикатор трекинга), экран оператора (метрики в реальном времени, диагностическая сводка).
- **Storage** — сохранение сессии: временные ряды ключевых точек, метрики, баллы, рекомендация.
- **Integration API** — передача результата основной программе АПК.

**Ключевые сущности данных:**

- `Patient` — внешний идентификатор (без ФИО внутри модуля).
- `TestSession` — дата, рука, параметры камеры, список заданий.
- `CalibrationProfile` — `palmWidth`, `palmCenter`, базовые нормированные расстояния.
- `TrackingFrame` — временная метка, 21 ключевая точка, флаг достоверности кадра.
- `ExerciseResult` — задание, статус (выполнено / частично / не оценено), частный балл, ключевые метрики.
- `TestSummary` — `validTrackingRatio`, блочные баллы, итоговый индекс, рекомендация.

**Результат:** архитектурная схема слоёв и сущностей, согласованная с командой.

**Длительность:** 1 неделя.

## Пункт 4. Проектирование метрик и алгоритма оценки

**Цель:** формализовать все вычисления до начала кодирования, чтобы реализация была однозначной.

**Работы:**

- Описать формулы для каждой метрики:
  - `validTrackingRatio = N_valid / N_total`
  - `normalizedDistance = distance / palmWidth`
  - `thumbIndexDistance / palmWidth` — порог щипка: идеал < 0.15, минимум < 0.25
  - `tipToPalmDistance / palmWidth` — порог кулака и открытой ладони
  - `fingerCurl` — классификация пальца: разогнут / частично согнут / согнут
  - `holdTime` — минимальное удержание: идеал 2 с, минимум 1–1.5 с
  - `handJitter = std(palmCenter за окно N кадров)` — порог стабильности
- Описать правила классификации каждого из 8 заданий (условия `true/false` + `"частично"`).
- Зафиксировать таблицу весов индекса готовности (из методики):

| Блок | Максимум баллов |
|---|---|
| Качество трекинга | 20 |
| Открытая ладонь | 10 |
| Кулак | 15 |
| Щипковый захват | 15 |
| Указательный жест | 10 |
| Поворот кисти | 10 |
| Перемещение по зонам | 15 |
| Стабильность удержания | 5 |
| **Итого** | **100** |

- Описать алгоритм принятия решения:
  ```
  if validTrackingRatio < 0.65 → "Повторить тестирование"
  elif score >= 80 → "Стандартный VR-сценарий"
  elif score >= 60 → "Адаптированный режим"
  elif score >= 40 → "Тренировочный режим"
  else → "Индивидуальная настройка / подготовительный этап"
  ```
- Добавить метрику симметричности: при наличии данных обеих рук сравнивать индексы готовности правой и левой руки.

**Результат:** документ с формулами, порогами и правилами классификации, согласованный с реабилитологом.

**Длительность:** 1 неделя (параллельно с пунктом 3).

## Пункт 9. Тестирование и пилотная эксплуатация

**Цель:** убедиться в корректности и устойчивости модуля перед вводом в эксплуатацию.

**Работы:**

- Функциональное тестирование:
  - Все 8 заданий выполняются корректно в штатных условиях.
  - Прерывание теста на любом задании не приводит к ошибке.
  - Классификация каждого задания совпадает с ожидаемым результатом на контрольной выборке.
- Тестирование устойчивости трекинга:
  - Нормальное освещение: `validTrackingRatio` ≥ 0.80.
  - Слабое освещение: модуль выводит предупреждение и предлагает улучшить условия.

  - Украшения на руке: модуль не падает, результат помечается при снижении `validTrackingRatio`.

  - Пациент с выраженными двигательными ограничениями (медленные движения, малая амплитуда).

- Валидация рекомендаций с реабилитологом: сравнение автоматической рекомендации с экспертной оценкой на 20+ сессиях.
- Пилотная эксплуатация: минимум 10 сессий с реальными пользователями, сбор замечаний.
- Логирование: каждая сессия сохраняет полный лог ключевых точек, метрик и ошибок трекинга.

**Критерии готовности:**
- Критических ошибок нет.
- Совпадение автоматической рекомендации с экспертной оценкой ≥ 80%.
- Список замечаний пилотной эксплуатации закрыт или запланирован.
