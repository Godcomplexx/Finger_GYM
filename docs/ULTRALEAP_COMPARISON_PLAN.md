# План разработки и сравнения версий MediaPipe и Ultraleap

## Статус реализации

В проект добавлен первый рабочий вариант версии B:

- общий интерфейс трекера: `src/tracking/base.py`;
- фабрика выбора источника: `src/tracking/factory.py`;
- режим MediaPipe: `python main.py --tracker mediapipe`;
- режим Ultraleap: `python main.py --tracker ultraleap`;
- обычный запуск `python main.py` показывает экран выбора трекера в интерфейсе;
- адаптер Ultraleap: `src/tracking/ultraleap_adapter.py`;
- сохранение источника трекинга в JSON-протоколе: поле `module.trackingSource`.

Версия Ultraleap получает события напрямую из `LeapC.dll` через `ctypes`, сопоставляет суставы Ultraleap с 21-точечной схемой текущего scoring engine и нормализует координаты из миллиметров в `image_normalized`. Перед сравнительными испытаниями нужно проверить режим на реальном датчике Ultraleap и при необходимости откалибровать переменные `ULTRALEAP_WORKSPACE_WIDTH_MM`, `ULTRALEAP_WORKSPACE_HEIGHT_MM`, `ULTRALEAP_WORKSPACE_CENTER_Y_MM`.

## Цель

Разработать и сравнить две версии модуля Finger GYM:

- версия A: текущий режим на веб-камере и MediaPipe Hand Landmarker;
- версия B: режим на Ultraleap с получением 3D-координат кисти и пальцев.

Цель сравнения - определить, какой источник трекинга точнее, стабильнее и пригоднее для предварительной оценки мелкой моторики кисти в условиях эксплуатации АПК.

## Рабочая гипотеза

MediaPipe удобен как доступный режим без дополнительного оборудования, но его координата `z` является относительной и менее надежной для функциональной 3D-оценки. Ultraleap использует специализированный оптический hand tracking и через LeapC SDK предоставляет данные о руке, пальцах и суставах в 3D-пространстве. Поэтому ожидается, что Ultraleap даст более стабильную оценку движений пальцев, особенно для упражнений на сгибание, удержание, поворот кисти и амплитуду.

## Источники данных

### Версия A. MediaPipe

Вход:

- RGB-видеопоток с веб-камеры;
- 21 ключевая точка кисти;
- нормированные координаты `x`, `y`, относительная координата `z`;
- метка руки `Left` / `Right`.

Ограничение:

- глубина не является метрической 3D-координатой;
- результат зависит от освещения, фона и качества камеры;
- часть поворотов кисти определяется косвенно.

### Версия B. Ultraleap

Вход:

- кадры трекинга от Ultraleap Tracking Service;
- 3D-положение ладони, пальцев, костей и суставов;
- ориентации кисти и пальцев;
- служебные признаки качества трекинга, если доступны в SDK;
- временная метка кадра.

Ограничение:

- требуется отдельное устройство Ultraleap;
- рабочая зона ограничена положением датчика;
- возможны потери при сильной окклюзии пальцев;
- требуется установка Ultraleap Hand Tracking Software и SDK.

## Архитектурный план

### 1. Ввести общий интерфейс трекера

Нужно отделить упражнения от конкретной библиотеки трекинга.

Предлагаемый интерфейс:

```python
class HandTracker(Protocol):
    source_name: str
    source_version: str

    def process(self, frame: np.ndarray | None = None) -> TrackingFrame:
        ...

    def close(self) -> None:
        ...
```

Для MediaPipe входом остается кадр веб-камеры. Для Ultraleap кадр камеры не нужен, потому что данные приходят от сервиса трекинга.

### 2. Расширить модель данных

Текущий `Point2D` уже содержит `x`, `y`, `z`, но для корректной поддержки Ultraleap лучше ввести нейтральную модель:

```python
class CoordinateSystem(str, Enum):
    IMAGE_NORMALIZED = "image_normalized"
    ULTRALEAP_MM = "ultraleap_mm"
    NORMALIZED_3D = "normalized_3d"

@dataclass
class Point3D:
    x: float
    y: float
    z: float

@dataclass
class TrackingFrame:
    timestamp: float
    landmarks: list[Point3D]
    is_valid: bool
    hand_label: str = "Unknown"
    source: str = "unknown"
    coordinate_system: CoordinateSystem = CoordinateSystem.IMAGE_NORMALIZED
    confidence: float | None = None
```

Для совместимости на первом этапе можно оставить имя `Point2D`, но технически правильнее перейти на `Point3D`.

### 3. Сделать два адаптера

Файлы:

- `src/tracking/mediapipe_adapter.py`;
- `src/tracking/ultraleap_adapter.py`;
- `src/tracking/base.py`;
- `src/tracking/factory.py`.

Запуск:

```bash
python main.py --tracker mediapipe
python main.py --tracker ultraleap
```

Для сравнительных сессий:

```bash
python main.py --tracker both
```

Режим `both` нужен, если веб-камера и Ultraleap работают одновременно. Он позволит записать две оценки одной и той же попытки.

### 4. Нормализация координат

Чтобы упражнения можно было считать одним и тем же кодом, нужно привести координаты к общей системе:

- MediaPipe: нормированные координаты кадра `[0, 1]`;
- Ultraleap: миллиметры в координатной системе датчика;
- общий слой: нормализация на ширину ладони, центр ладони и ориентацию кисти.

Для метрик мелкой моторики лучше использовать не абсолютные координаты, а нормированные признаки:

- расстояние кончик пальца - основание пальца / ширина ладони;
- расстояние большой - указательный / ширина ладони;
- угол сгибания пальца по фалангам;
- угол нормали ладони относительно датчика;
- RMS-дрожание центра ладони;
- стабильность удержания позы;
- доля валидных кадров.

## Этапы разработки

### Этап 1. Подготовка Ultraleap

Работы:

- установить Ultraleap Hand Tracking Software;
- проверить работу датчика в Ultraleap Control Panel;
- установить LeapC SDK;
- проверить пример получения кадров трекинга;
- зафиксировать версию SDK, версию сервиса и модель устройства.

Результат:

- документированный стенд;
- скриншот/протокол проверки датчика;
- тестовый скрипт, печатающий 3D-координаты кисти.

### Этап 2. Прототип Ultraleap-адаптера

Работы:

- создать `UltraleapTrackingAdapter`;
- получать кадры руки из LeapC;
- сопоставить суставы Ultraleap с 21-точечной схемой проекта;
- возвращать `TrackingFrame`;
- сохранять `source="ultraleap"`;
- добавить обработку отсутствия руки и потери трекинга.

Результат:

- программа запускается с `--tracker ultraleap`;
- упражнения используют тот же scoring engine;
- JSON-протокол содержит источник трекинга.

### Этап 3. Приведение метрик к 3D

Работы:

- заменить 2D-расстояния на 3D-расстояния там, где источник поддерживает 3D;
- для Ultraleap считать углы сгибания пальцев по фалангам;
- для поворота ладони использовать нормаль ладони/ориентацию кисти, а не косвенный 2D-признак;
- для тремора считать RMS в миллиметрах и нормированный RMS;
- оставить MediaPipe-совместимый fallback.

Результат:

- общий набор метрик для двух трекеров;
- отдельные дополнительные 3D-метрики для Ultraleap.

### Этап 4. Синхронный сбор данных

Для честного сравнения нужно записывать одинаковые попытки:

- камера пишет MediaPipe-поток;
- Ultraleap пишет 3D-поток;
- оператор запускает одно упражнение;
- оба источника получают одинаковый `sessionId`, `exerciseId`, `attemptId`;
- эксперт оценивает выполнение независимо от результата программы.

Минимальный набор:

- 10-15 участников для инженерного пилота;
- 8 упражнений;
- 2 руки, если возможно;
- 2-3 повтора каждого упражнения;
- нормальные и ухудшенные условия.

Для клинической валидации объем выборки должен утверждаться отдельно.

### Этап 5. Сравнительные метрики

Сравнивать нужно не только итоговый балл, но и качество трекинга.

| Метрика | MediaPipe | Ultraleap | Что показывает |
|---|---|---|---|
| `validTrackingRatio` | да | да | доля кадров, пригодных для анализа |
| Потери трекинга за упражнение | да | да | устойчивость источника |
| Время до первого валидного кадра | да | да | скорость захвата руки |
| Частота кадров трекинга | да | да | стабильность realtime |
| RMS-дрожание ладони | 2D-нормированное | 3D мм + нормированное | стабильность удержания |
| Ошибка категории относительно эксперта | да | да | точность оценки |
| Совпадение рекомендации с экспертом | да | да | практическая пригодность |
| Ошибка амплитуды | косвенно | лучше в 3D | качество оценки движения |
| Ошибка поворота ладони | косвенно | по ориентации кисти | качество оценки пронации/супинации |
| Устойчивость к освещению | важно | менее важно | зависимость от внешней среды |
| Устойчивость к окклюзии | средняя | проверяется | потери при перекрытии пальцев |

### Этап 6. Критерии выбора

Ultraleap считается лучше, если на одинаковом наборе попыток:

- `validTrackingRatio` выше не менее чем на 10 процентных пунктов;
- меньше ложных отказов по качеству трекинга;
- выше совпадение категории качества с экспертной оценкой;
- лучше воспроизводимость итогового балла при повторных попытках;
- ниже ошибка в упражнениях на поворот кисти и сгибание пальцев;
- время обработки остается приемлемым для realtime;
- оператору проще получить валидную запись.

MediaPipe можно оставить как базовый режим, если:

- точность достаточно близка к Ultraleap;
- нет возможности использовать отдельный датчик;
- сценарий требует минимального оборудования.

## Минимальный отчет сравнения

Отчет должен включать:

- дату и условия испытаний;
- версии ПО и SDK;
- модель Ultraleap-устройства;
- параметры камеры;
- число участников, упражнений и попыток;
- таблицу метрик по каждому упражнению;
- сравнение с экспертной оценкой;
- список отказов и причин;
- вывод: оставить MediaPipe, перейти на Ultraleap или поддерживать оба режима.

## Риски

| Риск | Мера |
|---|---|
| Ultraleap плохо видит руку вне рабочей зоны | Разметить рабочую зону, добавить экран позиционирования |
| Пальцы перекрывают друг друга | Ввести тесты окклюзии и правила непригодности |
| Сложная интеграция LeapC с Python | Начать с минимального C/Python bridge или готовых Python bindings, затем стабилизировать |
| Метрики MediaPipe и Ultraleap несопоставимы | Нормировать признаки на ладонь и хранить source-specific metrics отдельно |
| Нельзя использовать старые пороги | Провести отдельную настройку порогов для Ultraleap |

## Первое практическое действие

Сначала нужно сделать маленький технический spike:

1. Подключить Ultraleap к ПК.
2. Установить Ultraleap Hand Tracking Software.
3. Проверить руку в Control Panel.
4. Запустить пример LeapC.
5. Сохранить 10 секунд сырых 3D-данных руки в JSON.
6. Сравнить частоту кадров, стабильность и наличие суставов пальцев.

После этого можно принимать решение о реализации полного `UltraleapTrackingAdapter`.
