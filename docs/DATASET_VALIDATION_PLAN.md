# План датасетов и валидации

## Цель

Подтвердить, что модуль корректно, воспроизводимо и надежно формирует метрики, категорию качества и рекомендацию в пределах заявленного назначения.

## Наборы данных

Минимально требуются:

- инженерный набор для регрессионного тестирования алгоритмов;
- внутренний валидационный набор для настройки порогов;
- внешний независимый тестовый набор без пересечения с обучающими/настроечными данными;
- набор отрицательных сценариев: плохое освещение, потеря трекинга, выход руки из кадра, украшения, фоновые объекты.

## Метаданные записи

- обезличенный код участника;
- возрастная группа;
- доминирующая рука;
- исследуемая рука;
- камера и разрешение;
- освещенность и фон;
- упражнение;
- экспертная оценка;
- причины исключения записи, если есть.

## Разметка

Для каждой записи фиксируются:

- идентификатор упражнения;
- пригодность записи;
- экспертная категория качества;
- ошибки выполнения;
- моменты потери трекинга;
- комментарии эксперта.

Разногласия экспертов должны разрешаться согласительной процедурой. Итоговая эталонная разметка хранится отдельно от результатов автоматического анализа.

## Валидационные слои

1. Аналитическая валидация: корректность трекинга, метрик, устойчивости и воспроизводимости.
2. Клиническая связь: обоснование связи метрик с функциональным состоянием кисти.
3. Клиническая валидация: совпадение автоматической оценки и экспертного решения на целевой выборке.

## Стартовые KPI

- `validTrackingRatio` в нормальных условиях: не ниже 0.80;
- отказ от интерпретации при `validTrackingRatio < 0.65`;
- совпадение категории с экспертной оценкой: стартовая цель не ниже 80%;
- время формирования результата после завершения сессии: не более 2 секунд на целевом ПК;
- внешний тестовый набор: 0 пересечений с внутренними наборами.

Численные KPI должны быть утверждены клиническим экспертом и подтверждены протоколами испытаний.
